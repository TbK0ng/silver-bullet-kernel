## Context

SBK 已形成较完整的命令面（`flow / install / greenfield / blueprint / intake / adapter / semantic / fleet / verify / policy` 等），但操作知识分散在脚本、配置和历史文档中。用户目标是直接拿 SBK 在不同项目场景落地，并逐步把开发执行权交给 SBK，因此文档必须提供“可执行动作 + 决策依据 + 故障恢复”的闭环。

约束：

- 指南内容必须以代码库真实命令与参数为准，不能依赖外部想当然描述。
- 读者包含非程序员协作者，术语可用但需要“先说人话再给术语”。
- 平台策略采用 Codex 主线，其他平台仅附录说明边界与替代路径。
- 发布策略采用 Stable 默认路径，Beta 能力仅在风险提示下使用。

## Goals / Non-Goals

**Goals:**

- 构建 `docs/practice/` 的分层实践套件，覆盖“引入 -> 首轮可运行 -> 场景执行 -> 治理运维 -> 差异附录”。
- 每篇文档都给出可直接执行的命令与可复用提示词模板。
- 对相近功能给出选择依据、优劣与禁用场景，降低误用概率。
- 覆盖当前仓库主线能力（含 `sbk` 命令族、门禁、指标、语义工具、并行与规格迁移入口）。

**Non-Goals:**

- 不修改任何 SBK 运行时代码逻辑。
- 不新增或调整 adapter/blueprint/release channel 契约。
- 不替代 OpenSpec 规范文档；实践指南面向执行，不重复规范原文。

## Decisions

1. 文档信息架构采用“导航 + 快速上手 + 绿地/存量实战 + 能力手册 + 辨析 + 提示词 + 治理 + 附录”。
原因：执行者通常先要“跑通”，再要“做对”，最后才是“做优”；该结构对应学习曲线。

2. 统一章节模板，强制包含“手动命令 / AI 提示词 / 产物与成功判定 / 失败恢复 / 何时不用”。
原因：避免只有原理没有动作，或只有动作没有验收。

3. 使用 Stable 主线路径，Beta 集中在附录并给出启用前检查单。
原因：生产默认要可预期，实验能力必须显式风险承担。

4. 相近能力比较使用“决策问题 + 推荐顺序 + 优劣 + 反例”格式。
原因：非程序员在工具选择时更需要“先判断”而不是“先背命令”。

5. 提示词手册以 Codex 终端协作为主，使用可复制模板并区分“你手动输入”和“让 AI 自动执行”。
原因：用户目标是由 SBK 接管开发，提示词质量直接影响执行一致性。

## Risks / Trade-offs

- [风险] 脚本参数未来变更导致指南过期 -> [缓解] 文档中明确“以 `scripts/*.ps1` 为准”，并在能力手册按脚本来源编写。
- [风险] 文档过长导致读者迷失 -> [缓解] 使用导航文档与“一页上手”做入口分流。
- [风险] Beta 误用进入生产 -> [缓解] 附录单独标注，并在主线流程只给 Stable 命令。
- [风险] 非程序员误解术语 -> [缓解] 先给业务化解释，再给术语和命令。
